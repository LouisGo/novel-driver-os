# 作者可读语言规范

Novel Driver OS 的默认输出对象是创作者，不是开发者。系统可以保留内部字段和命令，但默认展示必须先使用中文创作语言。

## 分层原则

- 作者默认看到：草案、方向、卖点、追读点、风险、下一步选择。
- 高级信息可以看到：是否已保存、是否待确认、会影响哪些资料。
- 调试和自动化才看到：字段名、状态值、文件路径、命令和 JSON。
- 创作类输出必须有作者态结构：状态头、核心锚点、内容模块、待确认开关、风险和下一步。

`--json` 输出保持机器可读，不做中文化。普通 CLI 输出、GUI、agent 对话和 Markdown 报告应优先使用作者可读语言。

## 作者态结构

面向作者的创作输出不能只是一段段自然语言。默认使用中文标签，把内容拆成可审查、可选择、可继续加工的模块。

推荐结构：

```md
【类型】世界观草案 / 创作方向草案 / 爽点设计草案
【状态】未确认
【入库】不会写入正史资料
【用途】供作者选择、修改，后续可生成待确认提案

【一句话锚点】
用一句话说明本次输出最核心的创作判断。

【已形成的草案】
【核心规则】...
【主角压力】...
【追读点】...

【需要你确认】
【调性】...
【能力规则】...

【风险】
【设定风险】...
【代入风险】...

【下一步】
...
```

要求：

- 标签使用中文方括号，例如 `【核心规则】`，不要使用英文内部字段当标题。
- 每个关键判断都要能被作者单独接受、修改或拒绝。
- “需要你确认”不超过 5 项，优先列真正会改变后续写法的开关。
- “风险”必须具体到故事效果，例如设定过密、主角代入弱、爽点来源不清。
- 结构化不等于输出 YAML、JSON 或字段清单；除非用户要求调试或落盘格式。

## 写作规则

- 不把英文内部字段当标题，例如 `premise_alchemy_report`、`detected_type`、`route_plan`。
- 可以保留“分类感”，但用中文名表达，例如“分类：创作方向”“状态：等待确认”。
- 默认不要输出 YAML、JSON 或字段清单；只有用户要求技术细节、调试、命令或文件结构时才展示。
- 命令可以作为“高级操作”出现，但正文解释必须先说明它对创作者意味着什么。
- 报告里必须先回答创作问题，再列系统记录。
