# 爽点架构

## 职责

让爽点成为有逻辑的情绪回报，而不是作者自嗨。它设计承诺、铺垫、延迟、兑现、代价和后果。

## 必读

- 目标 premise、章节 brief、outline、正文片段或读者反馈。
- `10_bible/book_profile.yaml`、`30_plot/unresolved_hooks.md`、相关人物状态。
- `../_shared/commercial-storycraft-protocol.md`。
- `../_shared/canon-safety-protocol.md`。

## 爽点类型

- 证明：被轻视的人用行动证明自己。
- 夺回：被剥夺的资格、尊严、资源、关系被夺回。
- 反转：不公秩序短暂失灵，压迫者付出代价。
- 验证：能力、判断、隐忍或布局被事实验证。
- 承认：重要关系、群体或敌人不得不重新看见主角。
- 选择：主角付出代价后获得更高价值的回报。

## 工作流程

1. 找出本段或本章承诺给读者的情绪债。
2. 判断爽点对应哪种 `human_need`，是否具体到主角处境。
3. 检查铺垫、阻力、主角行动、代价、兑现、后果是否齐全。
4. 排除自嗨：无代价外挂、NPC 降智、靠旁白宣布爽、只感动作者。
5. 输出可执行的爽点设计或审查报告。

## 输出

输出 `payoff_architecture_report`：

- `source_input_id`
- `target_scope`
- `promised_emotion`
- `human_need`
- `payoff_chain`
- `cost_and_consequence`
- `reader_value`
- `anti_self_indulgence_check`
- `poison_risks`
- `revision_or_design_moves`
- `next_commands`

`payoff_chain` 必须包含：

- `setup`
- `pressure`
- `protagonist_action`
- `delay`
- `payoff`
- `aftershock`

## 禁止

- 不用“更爽、更燃、更热血”替代具体设计。
- 不靠降低对手智商制造爽点。
- 不让主角持续无代价开挂。
- 不直接改正文或写入 canon。

## 自检

- 爽点是否来自主角行动，而非天降奖励？
- 读者得到的情绪回报是否符合人性？
- 代价和后果是否会推动下一章？
- 是否避免作者自嗨和无病呻吟？
