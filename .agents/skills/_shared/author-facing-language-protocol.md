# 作者可读语言协议

所有面向创作者的回答，都必须先使用中文创作语言，而不是内部字段、英文缩写或代码术语。

## 默认输出

- 用“草案、创作方向、故事卖点、追读点、风险、下一步”组织内容。
- 创作类输出必须先给作者态结构头，再展开正文：
  - `【类型】`：世界观草案 / 创作方向草案 / 爽点设计草案 / 情绪节奏草案 / 章节作战简报。
  - `【状态】`：未确认 / 等待确认 / 已写入正史资料。
  - `【入库】`：不会写入正史资料 / 需要你确认后才可写入。
  - `【用途】`：说明这份内容接下来能用来做什么。
- 正文必须用中文方括号标出关键模块，例如 `【一句话锚点】`、`【核心规则】`、`【主角压力】`、`【追读点】`、`【风险】`、`【需要你确认】`、`【下一步】`。
- 保留分类感，但使用中文显示名，例如“分类：创作方向”“状态：等待确认”。
- 若内容只是候选，直接说“这是草案，还没定”，不要默认输出 `candidate`、`proposal`。
- 若需要作者确认，直接说“需要你确认后才会进入长期资料”。

## 作者态结构模板

新书、世界观、题材、爽点、节奏和章节规划类回答默认使用这个顺序：

1. 状态头：`【类型】`、`【状态】`、`【入库】`、`【用途】`。
2. 一句话锚点：先用一句话钉住核心创作判断。
3. 内容模块：按任务拆成可审查的小块，不写成连续散文。
4. 待确认开关：列出作者必须拍板的分歧，不超过 5 个。
5. 风险：标出设定、爽点、代入、节奏或 AI 味风险。
6. 下一步：给出作者可直接选择或 agent 可继续执行的动作。

允许用简洁段落解释，但每个关键创作判断必须落在清晰中文标签下。结构化不是输出内部字段；不要把 YAML、JSON 或英文 schema 当作默认作者界面。

## 术语转换

- `premise` -> 创作方向
- `payoff` -> 爽点设计
- `emotion` -> 情绪节奏
- `brief` -> 章节作战简报
- `route` / `route_plan` -> 下一步建议
- `candidate` -> 草案
- `proposal` -> 待确认提案
- `canon` -> 正史资料
- `context` -> 后续写作用资料
- `input_id` -> 输入编号
- `source_input` -> 原始输入
- `detected_type` -> 输入分类
- `status` -> 状态

## 禁止

- 不用英文内部字段当一级标题。
- 不默认输出 YAML、JSON 或字段清单。
- 不把 agent 技能名、schema 字段名、文件路径、状态机当作主回答。
- 不让机制说明压过创作建议。

## 允许

- 用户明确要求技术细节、CLI、开源实现、调试或文件结构时，可以展示内部术语。
- 展示内部术语时，必须同时给出中文含义。
- 机器输出、落盘文件、`--json`、测试和 schema 可以保留内部字段。
