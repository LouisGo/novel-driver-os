# 叙事机制提醒协议

本协议用于把经典中文网文经验转成可审查的项目状态、期待账本和复盘提醒。它不是套路执行器，也不是固定 prompt。

## 定位

- 只提醒作者避免写浅，不强迫作品变标准模板。
- 只记录项目当前可执行机制，不把理论说明写进正文或 context。
- storycraft report 是过程产物，story_engine.yaml 是项目级参考，canon_registry.md 才是正史。
- AI 推断必须保留 origin、confidence、obligation_level，不能伪装成读者真实期待。

## 检查机制

- 核心情绪是否仍能反复生产。
- 主角底层欲望是否还实。
- 世界机器是否有资源、阶层、规则、禁忌和秘密。
- 读者期待是否被打开、触碰、延迟、兑现、变形或主动放弃。
- 爽点是否有代价、后果和主角行动。
- 留白是否被保护，不能把所有未解释都判成伪钩子。
- 经典套路是否被重新解释，而不是直接复刻。

## Promise 安全

- `author_explicit` 和 `text_explicit` 权重高，但仍需要确认才能成为长期参考。
- `genre_expectation` 是类型承诺，不等于具体章节债务。
- `ai_inference` 默认只能是 `speculative`。
- `reader_likely` 需要 weekly alignment 校准。
- 未兑现不是错误；可能是延迟、情绪回响、反转、替代、误导或作者主动放弃。

## Context 最小注入

context build 只能注入当前任务可执行的项目状态：

- 当前核心情绪。
- 启用中的故事引擎。
- 相关 open/risk promise。
- anti_genes。
- gene_drift_candidate。

禁止注入：

- 完整理论说明。
- 经典作品分析。
- 本协议全文。
- “经典网文应该如何”的泛化段落。

## 禁止

- 不把 report 当配置。
- 不把配置当 canon。
- 不把 canon 当最终正文。
- 不用男频/女频标签替代机制判断。
- 不让“每章都要爽点、每卷都要升级、伏笔都要明还”变成硬规则。
